# 项目路线图

本文档描述这个仓库在当前基线之上的后续演进方向。

## 当前基线

当前仓库已经具备一条可运行、可验证、可做实验的主链：

- 输入支持 `toy`、`text`、`fvecs`、`bvecs`、`bin`
- 构建主链已经收拢到 `PipeANN disk build -> Gorgeous partition -> Gorgeous relayout -> PQ artifacts`
- 默认主输出为 `gorgeous-native`，兼容输出为 `graphrep`
- native 索引加载以原始 relayout 产物为主，sidecar 仅用于辅助校验或兼容消费
- native full-precision payload 采用按需读取，而不是启动期整包内存化
- 搜索支持 `PQ` 与 `full` 两种近似距离路径
- CLI 已覆盖 `build / search / inspect / ground_truth / bench / bench_merge / bench_compare`
- 测试已覆盖构建、搜索、CLI 与集成路径

## 总体目标

项目后续应同时满足两条主线：

- 系统复现主线
  - 更接近原始 `PipeANN` 的搜索执行模型、I/O 模型与量化路径
  - 更接近原始 `Gorgeous` 的分区、relayout、页布局与搜索行为
- 工程可用主线
  - 可稳定处理真实数据集与真实实验流程
  - 可维护、可测试、可比较、可扩展

## 下一阶段重点

### Phase 1: 搜索内核深化

目标：把“已经能跑”的搜索链路推进为“更接近真实系统实现”的搜索内核。

重点事项：

- 重构候选池、访问集、scratch 空间与查询态对象
- 继续补齐更接近原始 `PipeANN` 的统计字段与执行状态
- 明确 `PQ`、`full`、后续量化器的统一接口边界
- 引入更稳定的查询级性能回归用例

完成标志：

- 搜索核心状态不再分散在多个局部结构里
- 查询路径的指标输出可以稳定对比不同实现
- 新增量化器或新 I/O 后端时，不需要再改动大量搜索主循环

### Phase 2: I/O 与执行模型升级

目标：把当前可工作的异步兼容路径，升级为更接近原始磁盘 ANN 系统的 I/O 抽象。

重点事项：

- 在现有 reader 之上补出更清晰的后端抽象
- 在当前 `pread/std::async` 与 `libaio` 基础上，评估引入 `io_uring`
- 保留 aligned / unaligned 双 offset 语义，持续向原始 `IORequest` 模型收拢
- 补页缓冲、批量提交、完成队列、预取等机制

完成标志：

- 搜索主循环不再依赖具体 I/O 实现细节
- 可以以最小改动切换多个 I/O 后端
- I/O 统计能够区分提交、完成、等待与 CPU 计算阶段

### Phase 3: 量化与近似距离扩展

目标：让当前已接入的 `PipeANN PQ` 路径成为统一量化框架的一部分，而不是单一路径特例。

重点事项：

- 继续收敛 `PQ` 路径的元数据、加载、默认路径推导与检视能力
- 为后续 `RaBitQ` 或其他量化器预留统一接口
- 分离近似候选筛选、精确重排与 payload 访问策略
- 增加量化器相关 benchmark 与正确性校验

完成标志：

- 量化器的接入点清晰，搜索主循环无需感知具体实现细节
- 不同量化器可以共享实验导出和统计框架

### Phase 4: 构建链路继续贴近原始系统

目标：让项目内构建链路在保持统一入口的同时，继续向原始 `PipeANN/Gorgeous` workflow 收敛。

重点事项：

- 继续减少项目层对原始构建步骤的额外包裹
- 更完整地暴露 workflow 级 artifact 与 manifest
- 明确 native 输出与兼容输出的边界，避免兼容路径反向污染主语义
- 逐步补齐更多原始构建参数与命名语义

完成标志：

- 构建输出更容易被外部脚本、实验流程和检视工具直接消费
- native 产物自身能够承载主要语义，sidecar 只保留兼容和调试价值

### Phase 5: 实验体系与可比性增强

目标：把当前 bench 工具升级为更稳定、更适合连续实验的结果体系。

重点事项：

- 补充真实数据集的建议流程与标准参数模板
- 稳定实验目录、manifest、summary 字段
- 增加 recall、QPS、延迟之外的更多辅助指标
- 让 merge / compare 对多轮实验和多组 sweep 更友好

完成标志：

- 结果导出格式在多轮实验中保持稳定
- 同一数据集和同一参数集可以方便复现与对比

### Phase 6: 工程与仓库整理

目标：降低长期维护成本，让仓库结构与模块边界更清晰。

重点事项：

- 按模块收敛头文件与实现边界
- 减少历史命名、兼容层和临时适配逻辑带来的阅读负担
- 继续完善测试结构与文档结构
- 逐步补 CI 与更明确的开发约束

完成标志：

- 新贡献者可以较快定位构建、搜索、I/O、量化、工具链相关代码
- 仓库中的“当前主语义”和“兼容/历史路径”界限明确

## 推荐优先顺序

基于当前状态，建议优先推进：

1. 搜索内核深化
2. I/O 与执行模型升级
3. 量化与近似距离扩展
4. 构建链路继续贴近原始系统
5. 实验体系与可比性增强
6. 工程与仓库整理

## 近期里程碑

建议把最近一轮工作拆成以下里程碑：

1. `M1`
   - 收拢搜索查询态、候选池和统计结构
2. `M2`
   - 明确 I/O 后端接口，梳理当前 reader 分层
3. `M3`
   - 继续统一 `PQ` 路径与默认 artifact 推导
4. `M4`
   - 补实验指标与 benchmark 导出字段
5. `M5`
   - 补仓库结构整理与文档同步

## 成功标准

当下一阶段完成时，仓库应至少满足：

- 主链语义稳定：native 输出、payload 访问、workflow artifact 语义不再频繁变动
- 搜索内核可扩展：更换 I/O 后端或量化器时改动面可控
- 实验结果可比较：同一数据集和参数集的结果结构稳定
- 工程边界清晰：主实现、兼容路径、第三方 vendored 代码职责明确
