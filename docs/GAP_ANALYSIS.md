# 能力差距分析

本文档不再把当前仓库与 `PipeANN`、`Gorgeous` 的差距简单描述成“还没有 CLI / 还没有测试 / 还没有真实数据链路”。这些基础能力现在已经具备。本文档关注的是：在当前实现已经跑通主链之后，距离“更接近原始系统”还差哪些关键能力。

## 当前能力基线

当前仓库已经有这些能力：

- 支持真实数据输入：`text`、`fvecs`、`bvecs`、`bin`
- 支持统一构建链路：`PipeANN disk build -> Gorgeous partition -> relayout -> PQ artifacts`
- 支持两类索引消费：`gorgeous-native` 与 `graphrep` 兼容导出
- 支持 native payload 按需访问
- 支持 `build / search / inspect / ground_truth / bench / merge / compare`
- 支持构建、搜索、CLI、集成测试

因此，当前差距主要集中在“深度”和“逼真度”，而不再是“有没有这条能力”。

## 与 PipeANN 的差距

### 1. 搜索执行模型

当前已具备：

- 候选集与访问集
- 页请求派发与完成轮询
- 页驻留后的邻接扩张
- `PQ` 与 `full` 两条近似距离路径
- 查询级统计输出

仍有差距：

- 更接近原始 `PipeANN` 的 scratch / query buffer 复用模型
- 更成熟的候选池裁剪、插入和终止条件
- 多查询并行执行模型
- 过滤搜索、范围搜索、OOD 搜索等更多查询模式
- 更完整的原始统计字段与调优指标

结论：

- 搜索主循环已经存在，但距离“原始 PipeANN 级别的执行内核”还有一段距离。

### 2. I/O 模型

当前已具备：

- 页级读取抽象
- `libaio` 可选路径
- `pread + std::async` 兼容路径
- 向原始 `IORequest` 语义收拢的 aligned / unaligned offset 模型基础

仍有差距：

- 更独立的 I/O 后端分层
- `io_uring` 后端
- 更成熟的批量提交、完成队列与缓冲管理
- 更细粒度的 I/O 时延观测
- 多设备或更复杂底层后端支持

结论：

- 当前 I/O 足以工作和做实验，但还不够接近原始磁盘 ANN 系统的执行深度。

### 3. 量化与近似距离

当前已具备：

- PipeANN PQ artifacts 生成
- PQ metadata 加载与消费
- 全精度 fallback

仍有差距：

- 更统一的量化器抽象
- 更完整的 PQ 查表与调优能力整理
- `RaBitQ` 等其他量化器接入
- 更清晰的近似筛选与精确重排分层



### 4. 构建生态

当前已具备：

- 统一建盘入口
- 原始 PipeANN build 接入
- workflow 级 artifact 暴露
- PQ artifacts 生成

仍有差距：

- 更完整的原始构建参数语义覆盖
- 更强的外部脚本集成体验
- Python / 脚本层封装
- 更系统的数据集准备与批处理工具

结论：

- 构建主链已经打通，但围绕这条主链的生态工具还可以继续补强。

## 与 Gorgeous 的差距

### 1. 分区与 relayout 语义

当前已具备：

- 原始 `Gorgeous` 分区与 relayout 接入
- native relayout 元数据写出与加载
- native 产物优先的装载语义

仍有差距：

- 更完整地覆盖原始 `Gorgeous` 不同模式与配置
- 更精细的布局策略与元数据表达
- 更系统的 relayout 结果分析工具


### 2. 图复制搜索行为

当前已具备：

- 页内布局节点与邻接副本消费
- native 页面加载后的图扩张
- payload 按需精排

仍有差距：

- 更接近原始 Gorgeous page search 的细粒度行为
- 热页、热门节点、embedding cache 等缓存策略
- 更复杂的页级扩张控制
- 更系统的 duplicate graph 搜索策略

结论：

- 当前搜索已经利用了图复制布局，但还没有把 `Gorgeous` 的缓存和页级策略充分做深。

### 3. 工具与实验体系

当前已具备：

- inspect、bench、ground truth、merge、compare 工具链
- 实验目录与 manifest 导出

仍有差距：

- 更丰富的 relayout 检查工具
- 更体系化的数据转换与数据集脚本
- 更稳定的多轮实验基线

结论：

- 现在不是“没有实验工具”，而是“实验工具已经够用，但还需要继续结构化和扩展”。

## 工程层差距

当前已具备：

- CMake 构建
- 多个独立 CLI
- 单元/集成级测试目标
- README 与 docs

仍有差距：

- 更清晰的模块目录边界
- 更完善的 CI
- 更系统的日志与诊断体系
- 更少的历史兼容逻辑与命名负担
- 更稳定的开发约束与代码组织方式

结论：

- 工程基础已经存在，但距离“长期维护成本低、结构非常清晰”的状态仍有整理空间。

## 最关键的剩余问题

如果只看当前最影响后续演进的几个问题，优先级大致如下：

1. 搜索内核与查询态结构仍可继续收敛
2. I/O 抽象尚未完全独立，难以自然演进到 `io_uring`
3. 量化路径已经接入，但统一抽象还不够彻底
4. 构建 workflow 已接近原始系统，但生态工具还不够厚
5. 仓库结构与兼容路径仍需要继续清理

## 当前定位

一句话总结：

当前仓库已经是“具备完整主链与实验工具的 PipeANN x Gorgeous 集成工程”，但距离“在执行深度、I/O 模型、量化体系和工程整理上全面接近原始系统”的目标，仍有若干关键阶段要走。
